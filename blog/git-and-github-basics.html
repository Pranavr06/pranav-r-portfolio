Mastering the Basics of Git & GitHub

Version Control October 20, 2025 By Pranav R

Git and GitHub are essential tools for developers to manage code versions and collaborate on projects. This beginner’s guide covers the core concepts and commands to get you started with version control.

What is Git?

Git is a distributed version control system that tracks changes in your codebase, allowing multiple developers to work on the same project without conflicts.

Why Use GitHub?

GitHub is a platform that hosts Git repositories, making it easy to share code, collaborate with others, and contribute to open-source projects.

Git vs. GitHub: What's the Difference?

This is a common point of confusion. Think of it this way: Git is the tool you install on your computer to track changes, while GitHub is the website where you store your projects and collaborate. You can use Git without GitHub, but you can't use GitHub without Git.

Getting Started with Git

Here are the basic steps to set up and use Git:

First-Time Setup: Configure Git

Before you start using Git, you need to configure it with your name and email address. This information is attached to every commit you make. Open your terminal and run these two commands:

git config --global user.name "Your Name"
git config --global user.email "youremail@example.com"

1. Initialize a Repository

Navigate to your project folder in the terminal and run this command to create a new Git repository. This creates a hidden .git directory where Git stores all its tracking information.

git init

2. Add and Commit Changes

Before you save changes, you must add them to the "staging area." This lets you choose which modifications to include in your next save point (commit). Use git add . to stage all files, or specify a single file.

git add .
git commit -m "Initial commit"

3. Connect to GitHub

After creating a new repository on GitHub, link your local repository to it and "push" your commits to the remote server:

git remote add origin https://github.com/username/repository.git
git push -u origin main

4. Cloning an Existing Repository

If you want to work on a project that already exists on GitHub, you don't need to initialize a new repository. Instead, you "clone" it. This command downloads a full copy of the project to your computer, including all its history.

git clone https://github.com/username/repository.git

Basic Git Workflows

A "branch" is an independent line of development. It's best practice to create a new branch for every new feature or bug fix to keep your main branch clean.

git checkout -b feature-branch
# Make changes, then commit
git add .
git commit -m "Add new feature"
git push origin feature-branch

More Essential Commands

  • git status: Shows the current state of your repository, including which files are modified, staged, or untracked. Run this command often!
  • git log: Displays a history of all the commits you have made.
  • git pull: Fetches changes from the remote repository (like GitHub) and merges them into your current branch. This is how you stay up-to-date with collaborators.

Undoing Mistakes: Your Safety Net

One of the most powerful features of Git is the ability to undo mistakes. Here are a few common scenarios:

  • Discarding Local Changes: If you've made changes to a file but want to revert it back to how it was in the last commit, use: 
    git checkout -- <file-name>
  • Unstaging a File: If you've added a file to the staging area (with git add) but didn't mean to, you can unstage it with: 
    git reset HEAD <file-name>
  • Amending the Last Commit: If you made a typo in your last commit message or forgot to add a file, you can fix it with: 
    git commit --amend

A Note on Merge Conflicts

Sometimes, when you pull changes or merge branches, Git can't automatically combine the changes. This is called a "merge conflict." It happens when the same lines of code were changed in different ways. Don't panic! Your code editor will show you exactly where the conflicts are, and you simply need to choose which version of the code to keep.

Collaborating with Pull Requests

A Pull Request (or PR) is the heart of collaboration on GitHub. After you've pushed your feature branch to the repository, you can open a Pull Request. This is a formal way of saying, "I've completed my work, please review it and merge it into the main branch."

This process allows your teammates to review your code, leave comments, suggest changes, and run automated checks before the code is integrated. It's a crucial step for maintaining code quality in a team environment.

Best Practices

  • Commit Often: Make small, frequent commits with clear messages.
  • Use .gitignore: Create a .gitignore file in your project's root directory to tell Git which files and folders to ignore. This is crucial for excluding sensitive information (like .env files) or large dependency folders (like node_modules). 
    # .gitignore example
node_modules/
.env
*.log
  • Learn Pull Requests: Use GitHub’s pull requests to review and merge code.
